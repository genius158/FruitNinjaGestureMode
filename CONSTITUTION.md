# 编码通用法则（宪法）v1.0

## 一、核心原则

- `第一条：正确性优先`  
  代码必须先“做对”再“做快”；任何优化不得改变既定行为与业务语义。

- `第二条：安全性优先于便利性`  
  默认最小权限、输入必校验、敏感数据最小暴露；不以“开发方便”为由削弱安全。

- `第三条：可读性优先于炫技`  
  代码应让团队未来成员在最短时间读懂；命名清晰、结构清楚、意图明确。

- `第四条：可维护性优先于短期速度`  
  拒绝一次性“临时解法”长期化；重复逻辑抽象、边界明确、依赖可控。

- `第五条：简洁性原则`  
  在满足需求前提下选择最简单可行方案；不提前引入复杂架构或过度设计。

- `第六条：可测试性原则`  
  关键逻辑必须可验证；变更应伴随相应测试或可复现验证步骤。

- `第七条：兼容性与稳定性原则`  
  变更需评估对现有调用方、数据、接口和用户路径的影响，避免无告知破坏性修改。

- `第八条：性能与成本原则`  
  性能优化以度量为依据，优先解决真实瓶颈，避免凭感觉优化。

- `第九条：可观测性原则`  
  关键流程应可监控、可追踪、可告警，问题可定位、可回放、可恢复。

- `第十条：文档即交付物`  
  架构决策、接口约定、运行约束与已知风险必须留痕，且与代码同步更新。

## 二、优先级（冲突裁决顺序）

- `P0`：正确性 / 安全性 / 数据完整性
- `P1`：可维护性 / 可读性 / 可测试性
- `P2`：性能 / 成本 / 交付速度
- `P3`：风格偏好 / 个人习惯

当原则冲突时，始终按 `P0 -> P1 -> P2 -> P3` 决策，低优先级不得压过高优先级。

## 三、统一决策流程（3步）

- `步骤1：定义问题`：写清目标、约束、风险和影响面。
- `步骤2：列出选项`：至少两种方案，明确各自代价与收益。
- `步骤3：记录结论`：给出选择理由、回滚条件、复盘时间点。

## 四、例外条款（紧急场景）

允许先止血后完善，但必须满足：

1. 不新增高危安全风险
2. 可回滚
3. 有后续修复工单与截止时间

所有例外需在变更记录中说明“为什么例外、风险是什么、何时补齐”。

## 五、提交前通用自检（10条）

- [ ] 功能行为与需求一致，边界输入已考虑
- [ ] 无明显安全漏洞（注入、越权、泄露、硬编码密钥）
- [ ] 命名与结构清晰，代码无无意义复杂度
- [ ] 重复逻辑已收敛，模块边界清楚
- [ ] 有对应测试或可复现验证步骤
- [ ] 关键路径有日志/指标/错误信息可定位
- [ ] 对兼容性影响已评估并告知
- [ ] 性能变更有基本度量依据
- [ ] 文档/注释/接口说明已同步
- [ ] 回滚方案明确且可执行
