# spec.merge_prd.agent

## 1. 目标

将补充需求的变更记录（`additional`）合并进主 PRD（`mainprd`），输出一份语义一致、结构清晰、可继续用于后续任务拆解的主需求文档。

一句话定位：`spec.merge_prd.agent` 负责把增量需求改动“吸收”到主 PRD，而不是简单追加粘贴。

## 2. 使用方式

- 命令格式：`/spec.merge_prd.agent <mainprd> <additional>`
- 使用示例：`/spec.merge_prd.agent feature/PROJECT_SUMMARY.md spec_MODIFY_CONTROL_ENTRY_EXIT_TIME_task/CHANGED_LOG.md`

参数说明：

- `mainprd`：主 PRD 文件路径（被更新目标）。
- `additional`：补充改动来源文件路径（通常是 `CHANGED_LOG.md`，也可为补充 PRD 文档）。

## 3. 输入与输出

- 输入：
  1. 主 PRD 文件 `mainprd`。
  2. 补充改动文件 `additional`。
- 输出：
  1. 已融合后的 `mainprd`（原路径覆盖更新）。
  2. 一份本次合并摘要（包含“合并了什么 / 未合并什么 / 原因”）。
  3. 待确认项（若存在歧义）。

## 4. 合并原则（强制）

1. **语义合并优先**：按功能点、约束、验收标准合并，不做机械拼接。
2. **主文档一致性优先**：新增内容要融入现有章节结构与术语体系，避免重复章节。
3. **不丢信息**：`additional` 中对实现有约束意义的信息必须保留（可重写表述，但不可丢失约束）。
4. **可执行性优先**：合并后的 PRD 需可直接供 `/spec.task_prd.agent` 继续拆解任务。
5. **最小改动原则**：仅修改与补充需求相关的段落，保持未受影响章节稳定。

## 5. 执行流程

### Step 1：预检查

1. 校验 `mainprd` 与 `additional` 均存在且可读取。
2. 识别 `additional` 类型：
   - 变更日志类（如 `CHANGED_LOG.md`）。
   - 补充 PRD 类（结构化需求文档）。
3. 若任一输入为空、不可读或无法解析，进入阻塞（见第 8 节）。

### Step 2：抽取补充变更点

从 `additional` 中提取可落地需求信息，至少包含：

- 目标变化（新增/调整/删除）。
- 功能行为变化。
- 参数、时序、阈值、开关等约束变化。
- 影响范围与兼容性要求。
- 验收口径变化。

提取结果按“变更点”编号（如 `C1/C2/C3`）形成临时映射。

### Step 3：主 PRD 对位映射

1. 在 `mainprd` 中为每个变更点寻找对应位置：目标、功能、流程、约束、验收、风险。
2. 找到对应位置：执行“就地改写/补充”。
3. 找不到对应位置：新增最小必要小节（避免新增大而空的章节）。

### Step 4：冲突处理与决策

当 `mainprd` 与 `additional` 存在冲突时，按优先级处理：

1. 明确标注为“补充任务最新决策”的内容优先。
2. 与当前代码行为已对齐的内容优先。
3. 无法判定时保留主 PRD 原文，并在“待确认项”中列出冲突点。

### Step 5：回写主 PRD

1. 将合并结果写回 `mainprd` 原文件。
2. 确保结构连续、标题层级正确、术语前后一致。
3. 删除因合并产生的重复描述与互斥表述。

### Step 6：输出合并摘要

输出最少包含：

- 输入文件：`mainprd`、`additional`。
- 已合并变更点列表（`C1..Cn` + 对应落点章节）。
- 未合并项与原因。
- 待确认问题（若有）。

## 6. 推荐合并策略（CHANGED_LOG 场景）

当 `additional` 为 `CHANGED_LOG.md` 时，推荐按以下策略融合：

1. 将日志条目中的“完成动作”转为 PRD 的“目标/功能/约束/验收”描述。
2. 将“实现结论”转为 PRD 的“当前版本口径”或“版本说明”内容。
3. 过滤纯过程性描述（如“已执行某命令”），仅保留需求语义。
4. 若一条日志覆盖多项改动，拆分后分别合并到对应章节。

### additional 为 prompt 结构化文档时的映射规则

当 `additional` 为 `spec_additional/...md` 这类由 prompt 结构化生成的补充文档时，按以下规则映射：

1. `需求意图` -> `mainprd` 的“目标/范围”章节，作为本轮增量目标或约束更新。
2. `影响范围` -> `mainprd` 的“功能点/模块设计/接口影响”章节，明确受影响模块与边界。
3. `验收标准` -> `mainprd` 的“验收标准/测试口径”章节，作为后续任务拆解与验收依据。
4. 文档中的“兼容性与回滚说明”（若有）-> `mainprd` 的“风险与兼容性”章节，保留版本行为差异。
5. 若 `mainprd` 缺少对应章节，新增最小必要小节并保持章节层级与术语一致。

## 7. 产物质量标准

合并后的 `mainprd` 必须满足：

1. **完整性**：补充需求关键约束无遗漏。
2. **一致性**：不存在同一概念的冲突定义。
3. **可读性**：章节结构清晰，读者无需对照多文件理解需求。
4. **可拆解性**：可直接输入 `/spec.task_prd.agent` 生成任务计划。

## 8. 歧义与阻塞处理（强制）

出现以下情况必须暂停，禁止猜测合并：

1. `mainprd` 或 `additional` 文件不存在。
2. `additional` 仅有结论性语句，缺乏可映射的需求语义。
3. 同一变更点在 `additional` 内部自相矛盾。
4. 与主 PRD 关键约束冲突且无法判断优先级。

暂停时必须输出：

1. 阻塞原因（明确到文件与段落/条目）。
2. 最小必要补充清单（按优先级）。
3. 可直接回复的澄清问题示例。

## 9. 输出格式

每轮执行后，输出：

```markdown
## 合并摘要

- 输入主 PRD：`<mainprd>`
- 输入补充文件：`<additional>`
- 变更点提取：`C1..Cn`
- 已合并：`<C编号 -> mainprd章节路径>`
- 未合并：`<C编号 + 原因>`（若有）
- 冲突处理：`已解决/待确认`（含依据）
- 输出文件：`<mainprd>`
- 待确认问题：（若有）
```

要求：结论明确、路径可追踪、可直接用于后续任务拆解与实现。
