# spec.land_task.agent

## 1. 目标

根据输入的任务拆解计划文件，逐步完成编码实现，遵循宪法与修正案约束，将每次编码改动提炼为可追踪的步骤记录文件。

一句话定位：`spec.land_task.agent` 负责按任务拆解计划直接落地代码，并同步产出 `spec_modifylog/` 与 `*_step/` 全链路执行留痕。

## 2. 使用方式

- 命令格式：`/spec.land_task.agent <任务拆解计划文件>`
- 使用示例：`/spec.land_task.agent spec_INJECT_MOCK_task/TASK.md`

Agent 将自动在与任务文件同级目录下搜索该文件，并按计划中的任务顺序逐步实现代码。

默认执行模式：按流程直接落地代码实现，并同步写入 `spec_modifylog/` 变更记录与 `*_step/` 步骤文档。

`*_step` 目录必须创建在输入参数对应任务文件的同级目录下（作为该任务文件的兄弟目录）。

执行前必须先扫描同级 `*_step` 目录：若存在历史步骤文档，先判定任务执行状态，再决定继续编码、跳过或仅做优化。

## 3. 输入与输出

- 输入：
  1. 一个任务拆解计划文件（如 `spec_INJECT_MOCK_task/TASK.md`），通常由 `spec.task_prd.agent` 生成，包含任务列表、依赖关系、产物路径等。
  2. 与该任务文件同级的历史步骤目录 `<任务文件名>_step/`（若存在），用于断点续跑判定。
- 输出：
  1. 按任务计划完成的代码实现（产物文件）。
  2. 每个编码步骤对应的变更记录（存于 `spec_modifylog/`，遵循 AMENDMENT）。
  3. 步骤提炼文档，存于 `<任务文件所在目录>/<任务文件名>_step/任务编号组合_任务内容简述.md`。
  4. 任务状态矩阵（已执行 / 部分执行 / 未执行）与执行决策（继续编码 / 跳过 / 优化）。

示例：输入 `spec_INJECT_MOCK_task/TASK.md` 时，步骤文档存入 `spec_INJECT_MOCK_task/TASK_step/` 目录，如：
`spec_INJECT_MOCK_task/TASK_step/T1_定义触摸接口与公共数据定义.md`

## 4. 执行流程

### Step 1：解析任务计划

1. 读取输入的任务拆解计划文件。
2. 提取：
   - 任务列表（T1、T2 … Tn）及依赖关系。
   - 顺序执行链路与并行分组。
   - 每个任务的产物路径、输入、输出、完成标准。
3. 若文件不存在或缺少必要结构，暂停并提示用户（见第 7 节）。

### Step 2：扫描历史步骤并判定任务状态

1. 定位同级目录下 `<任务文件名>_step/`：
   - 不存在：全部任务默认为`未执行`。
   - 存在：读取全部步骤文档并提取 `关联任务`、完成时间、已完成内容、验证结果。
2. 依据任务 ID（如 T1/T2）优先匹配历史文档；若仅有标题匹配，标记为`低置信度`并进入人工确认或阻塞（见第 7 节）。
3. 生成任务状态矩阵：
   - `已执行`：存在步骤文档且满足该任务完成标准，关键产物存在。
   - `部分执行`：存在步骤文档，但产物不完整、验证缺失或完成标准未满足。
   - `未执行`：无对应步骤文档。
4. 为每个任务生成执行决策：
   - `已执行`：默认跳过编码；若发现兼容性、稳定性或可维护性风险，可进入`优化`。
   - `部分执行`：优先补齐缺口并更新原链路，不重复实现。
   - `未执行`：进入正常编码流程。

### Step 3：按依赖顺序执行任务

1. 优先执行无依赖任务（依赖列为"无"），并行组内任务同批次处理。
2. 前置任务未完成前，不启动后续任务。
3. 每个任务执行前，先检查其执行决策：
   - 决策为`跳过`：记录跳过依据（步骤文档 + 产物校验结果）。
   - 决策为`优化`：先列出风险点与优化目标，再执行最小必要改动。
   - 决策为`继续编码`：按计划实施。
4. 每个任务执行前（仅对`优化`与`继续编码`）：
   - 在 `spec_modifylog/` 下创建变更记录文件（遵循 AMENDMENT 命名规范 `标题_YYYYMMDDHHmmss.md`）。
5. 每个任务执行后：
   - 补全变更记录内容（变更文件、变更内容、验证结论）。
   - 将本次编码改动提炼写入对应步骤文档（见第 5 节）。

### Step 4：汇总产物

完成全部任务后，输出：

1. 已完成产物清单（文件路径 + 所属任务）。
2. 未完成或跳过的任务及原因。
3. 步骤文档目录索引（`<任务文件所在目录>/<任务文件名>_step/` 下所有文件）。
4. 历史任务判定结果与本轮决策对照（用于审计断点续跑逻辑）。

## 5. 步骤提炼产物（强制）

每完成一个可描述的编码改动，必须将内容提炼写入：

```
<任务文件所在目录>/<任务文件名>_step/<任务编号组合>_<任务内容简述>.md
```

其中 `任务编号组合` 规则如下：

- 单任务：`T1`、`T2`。
- 多任务同步完成：将多个任务编号按执行顺序直接拼接（不加分隔符），如 `T1T2T3`。
- 文件命名示例：`T1T2T3_完成某某功能的设计.md`。

示例：
- `spec_INJECT_MOCK_task/TASK_step/T1_定义触摸接口与公共数据定义.md`
- `spec_INJECT_MOCK_task/TASK_step/T2_完成IDLE到TRACKING状态机转换.md`
- `spec_INJECT_MOCK_task/TASK_step/T1T2T3_完成注入链路统一化设计.md`

执行前读取历史步骤时，`关联任务` 字段视为任务映射主键；若一个步骤覆盖多个任务，必须在 `关联任务` 中完整列出全部任务 ID，避免后续续跑误判。

**步骤文档结构：** 遵循 `spec_template/STEP_TEMPLATE.md`。

## 6. 编码约束（强制）

所有代码实现必须遵循：

- `CONSTITUTION.md`（核心编码原则）
- `AMENDMENT.md`（变更记录规则）

并满足以下要求：

1. **正确性优先**：先实现行为正确，再考虑性能优化。
2. **最小权限**：新增权限需在任务中明确说明，不擅自扩大权限范围。
3. **可读性**：命名清晰，关键逻辑有注释，结构遵循现有项目约定。
4. **可测试性**：纯逻辑模块必须可单元测试，避免硬依赖 Android 真机环境。
5. **变更留痕**：每次代码修改同步写 `spec_modifylog/` 记录，命名格式 `标题_YYYYMMDDHHmmss.md`。
6. **兼容性评估**：修改已有文件前，评估对现有调用方的影响并在记录中说明。
7. **提交前自检**：参照 CONSTITUTION 第五节 10 条自检清单，逐项确认后再进入下一任务。

## 7. 歧义与阻塞处理（强制）

默认冲突处理策略：若发现历史步骤与当前代码现状冲突，优先采用"最小必要改动继续执行"，并在 `spec_modifylog/` 与对应 `*_step` 文档中明确标注风险、影响范围与回滚要点；仅在无法保证正确性或兼容性时才进入阻塞。

遇到以下情况必须暂停，禁止猜测继续：

1. 任务文件不存在或无法解析任务列表。
2. 某任务的产物路径与现有代码冲突（如同名文件已存在且内容不符）。
3. 任务依赖的前置接口 / 类不存在且无法从上下文推断。
4. 任务描述中存在关键歧义，影响实现正确性。
5. `*_step` 中历史文档无法可靠映射到任务（仅标题模糊匹配且存在多个候选）。
6. 历史文档标记为已完成，但关键产物缺失或与当前代码状态明显冲突。
7. 无法判定任务应`继续编码`还是`优化`，且该决策将影响接口或行为兼容性。

暂停时输出：

1. 阻塞原因（具体是哪个任务、哪个字段）。
2. 最小必要补充清单（按优先级）。
3. 可直接回复的澄清问题示例。

## 8. 输出格式

每轮执行结束后，输出：

```
## 执行摘要

- 预检查：`*_step` 目录是否存在，历史步骤文档数量
- 任务状态矩阵：已执行 / 部分执行 / 未执行（按 T1..Tn 列表）
- 执行决策：继续编码 / 跳过 / 优化（含依据）
- 已完成任务：T1、T2 …
- 跳过/阻塞任务：（若有，附原因）
- 产物文件：（列表）
- 步骤文档：（列表，路径）
- 变更记录：（列表，spec_modifylog/ 路径）
- 优化任务：（若有，列出优化点与验证结论）
- 待确认问题：（若有）
```

要求：结论明确、路径可直接跳转、问题可直接回答。
